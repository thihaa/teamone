SSI Continued

Reminders about SSIs:

  • You need to use the *.shtml file extension (on this server) for your SSIs to work
  • You can only inlcude files from the current directory or subdirectories. In order to traverse to a parent directory, you must use: include virtual=”filename”
  • They are included on the server side. That means your browser only ever sees the results of the server included content.
  • What your pages look like on the server and what they look like on the client are different. That’s on purpose.
  • The basis for template development. Content can change on you, so you’ll need to adapt to that changing content

SSI Exercise: Gallery

This is an exercise that will demonstrate how you can get content from the server and allow the user to populate that content by selecting what they want to see.

Basically this is a poor-person’s Image Gallery, but it uses a Query String that tells the SSI to select a particular image.

SSI Query String Gallery

Using Server Side Includes (SSIs) and a Query String, you will create a gallery that exists as a single HTML page that will load a different image based on what is called in the Query String.

Build a new page on to add to your Brewery site that is called “gallery” and then select (at least) 3 images that are all approximately the same size.

You will create a menu that links to each of the images, or, if you are able to, find some thumbnails or icons.

Requirements

In order to do this correctly, you will need:

  • a server that supports SSIs. Most modern Web Servers have this, including the most common: Apache, Windows, Apple, IBM, etc.
  • HTML files will need to be processed by the SSI sub-system. Most of the time, this means saving them as *.shtml
  • You will need to have images that all have the same extension, such as JPG or GIF.
  • Not absolutely required, but thumbnail images for the user to click on would be useful. However, can be done with text links as well.

How to create the Gallery

Make sure all of your images are located in the same directory, and have “appropriate” names. This means: no capital letters, no spaces, no special characters.

Here are some acceptable examples:

  • image01.jpg
  • birthday_boy.jpg
  • sunshine.jpg
  • beer_ipa.jpg

Create the page you want the images to appear on. Example: gallery.shtml

In order to call a sepecific image, you will need to “pass” some data to the SSI engine in the URL. This is done by appending a question mark and the name of your image (without the extension) to the URL.

gallery.shtml?sunshine

The Query String in the previous example is now: sunshine

Template Development

While the concept is identical (The ability to merge HTML, CSS and Syntax together), the implementation is totally different: template development in a Content Management System.

By far the most common type of web development work done today: customizing templates or writing them from scratch

There are an incredible number of different languages and tools for developing templates, with Wordpress and Drupal being the most well known.

One common principle: inserting content into HTML

The concept on how it works is exactly like SSI: combining content from a server-side source before delivering documents to User Agents.

How do they work?

At their simplest, any CMS architecture will take content provided by the user (Generally a title and some content) and push it to a web space. Using templates, CMS software will take the content entered by the user, and insert it into the HTML where there are tags responsible for certain bits of information such as date, author, title, content, product name, cost, image and many other possibilities.

You can see that there is the CMS-specific {block} tag with an attribute of Description. That indicates that the content inside the tag will be related to the description as entered by the author. Also notable is the inclusion of regular HTML inside the Tumblr tags - exactly like our SSI exercises.

Once your template has been created, the CMS will scan your template for tags that indicate where data should be inserted. The application will then insert all of the relevant data and publish to completed HTML page to the specified server. The process works exactly like Server Side Includes, but the result resides on the web server (instead of in a browser window). This is an advantage, as further server processing can be done when a page is actually requested by a user.
